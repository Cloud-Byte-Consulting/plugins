# Maturity Survey Instrument (Increment I7)

## Use and limits

Use this survey to confirm system evidence and fill questions systems cannot
answer. It is an assessment instrument, not a psychometrically validated
industry benchmark and not an employee-performance tool. Survey results are
Attested evidence; they do not directly assign a platform stage, ASDLC level,
or Agentic Developer Portal gate.

Before distribution, record:

- assessment purpose, owner, population, and observation window;
- privacy/HR/legal and worker-representation approvals where applicable;
- invited count, response count, response rate, and cohort definitions;
- the approved minimum cohort size and suppression rule;
- retention/deletion date and who can access raw responses.

Distribute the same core questions separately to platform staff, platform users,
security/operations, and engineering leadership. Report team-level aggregates
and cohort differences; never expose individual responses or small groups.

## A. Platform investment and operating model

1. **[CONFIRM]** Which statement best describes platform ownership? *(No named
   owner / informal shared ownership / dedicated team / platform product with a
   funded roadmap and service commitments)*
2. **[CONFIRM]** How are platform priorities normally selected? *(Urgent request
   queue / leadership mandate / user research and demand evidence / measured
   product and business outcomes)*
3. **[GAP-FILL]** Which capability is most underfunded relative to current
   demand, and what work is delayed as a result?
4. **[GAP-FILL]** How confident are you that platform ownership, support, and
   escalation paths are clear? *(1–5 plus optional explanation)*

## B. Interfaces, self-service, and flow

1. **[CONFIRM]** For your last eligible environment or service request, how was
   it completed? *(Self-service without a person / standard request plus human
   approval or fulfillment / custom ticket or message / do not know)*
2. **[CONFIRM]** How often do supported golden paths meet the whole need without
   bypass or custom work? *(Never / rarely / about half / usually / always)*
3. **[GAP-FILL]** What is the most common reason people bypass the portal,
   catalog, template, or platform workflow?
4. **[GAP-FILL]** Which repeated manual request should become self-service next,
   and why?

## C. Measurement and delivery system

1. **[CONFIRM]** Which delivery and reliability outcomes are reviewed on a
   recurring cadence? *(Deployment frequency / lead time / change failure /
   recovery / platform reliability / developer experience / none / other)*
2. **[CONFIRM]** Do required tests, security checks, policy checks, promotion,
   rollback, and production observation use a consistent governed path?
   *(Never / for some workloads / for most / for all eligible workloads)*
3. **[GAP-FILL]** How much do practitioners trust the metrics used for platform
   and delivery decisions? *(1–5 plus optional explanation)*
4. **[GAP-FILL]** Describe one recent decision that changed—or could not be
   made—because of available or missing evidence.

## D. Agent governance and execution

1. **[CONFIRM]** Is there a current approved inventory of coding agents, agent
   identities, tools, and MCP integrations? *(Complete / partial / no / unsure)*
2. **[CONFIRM]** Can approved agent activity be attributed to a scoped,
   short-lived, revocable non-human identity from request through production?
   *(All paths / some paths / no / unobserved)*
3. **[GAP-FILL]** Where does uncertainty about approval, security, or ownership
   most often block responsible agent adoption?
4. **[GAP-FILL]** What agent action would you not permit today, and what evidence
   or control would change that decision?

## E. Agent context, skills, and reuse

1. **[CONFIRM]** How often can an approved agent find current build, test,
   ownership, architecture, and operational instructions without asking a
   person? *(Never / rarely / about half / usually / always)*
2. **[CONFIRM]** How are reusable skills, prompts, and tool configurations shared?
   *(Governed registry / maintained team library / informal messages or docs /
   not shared / not applicable)*
3. **[GAP-FILL]** Which missing or stale organizational context causes the most
   agent rework or human interruption?
4. **[GAP-FILL]** Name one reusable capability that exists but is difficult for
   other teams or agents to discover.

## F. Adoption, trust, and organizational constraints

1. **[CONFIRM]** Why do people use the platform today? *(Required by policy /
   useful for some tasks / preferred because it is easier and safer / users help
   improve it / use is inconsistent)*
2. **[CONFIRM]** How safe is it to challenge an automation result, report a
   failure, or request a human path? *(1–5 plus optional explanation)*
3. **[GAP-FILL]** What is the largest organizational—not technical—constraint on
   improving platform or agentic delivery maturity?
4. **[GAP-FILL]** What is one change that would increase your willingness to use
   or delegate work to the platform or an approved agent?

## Analysis contract

1. Preserve question wording, response options, survey dates, and cohort
   definitions so later cycles are comparable.
2. Report counts, denominators, response rates, suppression, and limitations.
3. Compare cohorts without averaging away disagreements.
4. Map responses to existing evidence IDs and label each finding confirmed,
   contradicted, or gap-filled.
5. Add stable evidence citations to the assessment state. Keep raw responses
   outside the shared state and delete them on the approved schedule.
6. Send reconciled evidence to the formal scorers. Never translate a response
   band directly into a maturity stage or combine the three assessment models.
